# Отчёт о первичном тестировании прототипа и соответствии плану НИР

**Программный комплекс:** EcomProfit Guard — веб-система анализа и прогнозирования рентабельности ecom-проектов.  
**Вид работ:** первичное тестирование прототипа, фиксация метрик и артефактов для задания **5П** (Трек П); сопоставление реализованного функционала с планом НИР.  
**Дата:** апрель 2026 г.

---

## Аннотация

В отчёте приведены результаты **модульного тестирования** backend (pytest), описана **методика проверки** модуля прогнозирования рентабельности (ARIMA, CatBoost, ансамбль), перечислены **рекомендуемые артефакты** для сдачи (скриншоты, примеры логов API). Отдельно дано **сравнение запланированного и реализованного** функционала с акцентом на улучшение прогнозирования и учёт **плановой и фактической** рентабельности. Числовые метрики качества ML на конкретной производственной выборке в прототипе **не выводятся в ответ API** (поле `metrics` пустое); их фиксация для отчёта возможна по методике, указанной в разделе 4.

## 4. Прогноз рентабельности (ML): методика и параметры

### 4.1. Реализованные модели

- **ARIMA:** подбор порядка по AIC, проверка стационарности (ADF), прогноз на горизонт.
- **CatBoost:** табличные признаки (лаги 1–3, скользящие средние по рентабельности и денежным рядам, тренды, календарь, синус/косинус месяца, годовые лаги 12/24); целевой сдвиг на один шаг вперёд; отложенная валидация на **последних до 6 точках** (`n_val = min(6, len(work)//3)`).
- **Ансамбль:** взвешенное усреднение прогнозов ARIMA и CatBoost (по 0.5 при доступности обеих веток), иначе используется доступная модель; при нехватке данных — **naive** (повтор последнего значения / упрощённый ряд).

### 4.2. Параметры из кода (справочно)

| Параметр | Значение в реализации |
|----------|------------------------|
| Максимальный горизонт | `min(запрос, 12)` месяцев |
| Обучение CatBoost | не раньше чем при `len(df) >= 12` для полного контура признаков (`MIN_HISTORY_FOR_FULL`) |
| Переход на naive | если месяцев меньше `forecast_min_months_history` (настройка `FORECAST_MIN_MONTHS_HISTORY`, по умолчанию в коде **6**), либо при ошибках модели |
| Окно обучения | последние до **36** месяцев (`MAX_TRAIN_MONTHS`) |
| Ограничение выхода | рентабельность на графике в диапазоне **[-100; 100]** п.п.; винзоризация обучающего ряда по перцентилям 5 и 95 % |

### 4.3. Метрики качества (MAE, MAPE) в отчёте

В текущем API ответ прогноза содержит `"metrics": {}` — **агрегированные ошибки на валидации в ответ не включаются**. Для заполнения разделов отчёта с численными MAE/MAPE возможны варианты:

- временное логирование ошибки на валидационной выборке CatBoost после `model.fit` в `forecast_ml.py`;
- offline-сравнение `y_pred` и факта на исторических месяцах (скрипт или notebook).

До выполнения такого прогона в таблице метрик допустима отметка «по данным полевой таблицы — не фиксировалось»; **методика верификации** при этом зафиксирована настоящим отчётом.

### 4.4. Сезонная корректировка

К прогнозу применяется постобработка `_apply_seasonal_variation`: добавка по среднему отклонению рентабельности по номеру месяца в истории (см. `forecast_ml.py`), что снижает «плоский» прогноз без учёта сезонности.

### 6.2. Прогнозирование: план и факт разработки

**План НИР** предполагал снижение ошибки прогноза за счёт комбинации моделей и учёта специфики ecom-данных, с перспективой сезонности и обратной связи.

**Фактически в прототипе:** реализованы ансамбль, богатый набор признаков для CatBoost, винзоризация и ограничение выхода, переход на naive при короткой истории, сезонная постобработка прогноза. Это соответствует направлению НИР; **полная** количественная оценка качества на реальных данных требует отдельного прогона с фиксацией MAE/MAPE (см. п. 4.3).

### 6.3. Плановая и фактическая рентабельность

- **Фактическая** рентабельность отображается на основе выручки и затрат — соответствует учётным данным.
- **Плановая** величина поддерживается на уровне импорта в модель `Act` (`profitability_plan` и синхронизация); **сравнение план/факт** как отдельный управленческий модуль в UI **не входил** в объём прототипа и отнесён к дорожной карте.

### 6.5. Прогнозирование: детализация

**Запланировано в НИР:** снизить ошибку прогноза за счёт комбинации моделей; заложить учёт сезонности и обратной связи по ошибке.

| Элемент | Описание |
|---------|----------|
| **Ансамбль** | Взвешенное усреднение прогнозов ARIMA и CatBoost при доступности обеих моделей |
| **ARIMA** | Подбор порядка по AIC, проверка стационарности (ADF), ограничение порядков на коротких рядах |
| **CatBoost** | Табличные признаки: лаги, скользящие средние по рентабельности и деньгам, тренды, календарь, годовые лаги |
| **Устойчивость** | Винзоризация при обучении; ограничение выхода прогноза; fallback на naive при нехватке данных |
| **Сезонность** | Признаки месяца и постобработка по истории (`forecast_ml.py`); полный SARIMA — возможный этап 2 |

**Запланировано далее:** явный **SARIMAX** с сезонным лагом 12 при достаточной истории; **персистентное хранение** прогнозов и расчёт ошибки по факту с коррекцией смещения; метрики MAPE по горизонту, доля попаданий в допуск.

### 6.6. Плановая и фактическая рентабельность (детали)

| Аспект | Статус |
|--------|--------|
| Импорт плана из таблицы (колонка в акте) | Поддержка на уровне модели `Act` и синхронизации |
| Отдельный экран «План vs факт» с отклонениями | **Не реализован** — в плане следующей итерации |
| Связь плана с прогнозом ML | В прототипе прогноз не привязан к бюджетному плану; сопоставление «ожидаемой» и фактической — задел |

**Вывод:** фактическая рентабельность и аналитика реализованы; сравнение с плановой как отдельный управленческий модуль — в дорожной карте (объём прототипа, приоритет ML-прогноза).

## 8. Выводы

1. **Автоматические тесты** backend по логике работы с Google Таблицами (нормализация, числа, периоды) **пройдены полностью** (6/6), что подтверждает корректность базового контура импорта данных для прототипа.  
2. **Модуль прогнозирования** реализует заявленный стек (ARIMA, CatBoost, ансамбль, защитные режимы); **численные метрики качества** на конкретной таблице заказчика в отчёт следует добавить после отдельного измерения или расширения логирования.  
3. **Соответствие плану НИР:** основной функционал аналитики и ML-прогноза **реализован**; **улучшение прогнозирования** — в части ансамбля и признаков; **план/факт** как отдельный интерфейс — **запланирован на следующий этап**, при этом данные плана заложены в модель.  
4. Для завершения комплекта **первичного тестирования** по заданию 5П необходимо приложить **скриншоты** по чек-листу и при необходимости **обезличенные логи** API; структура каталогов и пример JSON приведены в `docs/testing/`.
